Using swift modeller v2.0

Swift modeller

The screens of SWIFT MODELLER are designed in such a way that they depict the MODELLER protocol and steps of homology modeling. First time users of the GUI are greeted with a login screen where they would be required to enter the installation path of MODELLER. This screen will appear only once on a terminal until and unless the file containing the path is deleted.

The protein sequence input is taken alongwith a suitable five letter title to the project in the first screen of the GUI. The PIR database containing PDB sequences clustered at 95% identity has been downloaded and embedded in this software as an offline database. The first screen gives an option to the user to search templates for the target sequence in this offline database, alternatively user can upload previously downloaded sequences (after doing BLAST search of the target sequence) also.

On selecting the database search option the software edits a python file named “build_profile.py”, in the background and executes the same via MODELLER which results in creation of an output file named “build_profile.prf”, which contains search results. Our software extracts data from this file and presents it in a tabular and well readable form. The table contains template sequence codes, sequence identity percentage and E-value, the first column being a checkbox for selection of suitable templates. On selection of the relevant templates the system downloads these from the PDB website at the background and extracts these in the working directory. By default a single template is considered however, an option of using multiple templates is also available.

Login Screen of SWIFT MODELLER

These templates become input to another python file named “compare.py” which is executed and data (dendrogram) is extracted from a log file named “compare.log” and displayed. Radio buttons against each template is provided for easy selection on which the target sequence is to be modelled. The selection of the template is made on the basis of crystallographic resolution factor and phylogeny score. The target sequence is then aligned to the selected template via “align2d.py” and the alignment is then displayed. The user is asked to provide the number of models to be constructed on the basis of the target-template alignment.

The models are then constructed by the “model-single.py” (or “model_mult.py” depending upon the mode selected – single template or multiple templates). The DOPE assessment score of the built models is displayed alongwith GA341 and molpdf scores. Evaluation graphs are now plotted for each model which gives the comparison in the pseudo energy level of every amino acid of the built model and the template used. A series of python scripts are edited and executed for this purpose (“evaluate_model.py”, “evaluate_template.py” and “plot_profiles.py”). This plotted result is then framed and displayed with an option for loop modeling.

Loop Modelling is performed if the evaluation graph shows irregularities between the model and template’s plots in some pockets of amino acid residues. The user is required to enter the amino acid residue range and the number of loop models to be made, on selection of the loop modeling option. “loop_refine.py” performs loop modeling on the selected model in the denoted amino acid residue range. The assessment scores of the constructed loop models are displayed. The evaluation graphs for the newly constructed models can be plotted through this screen which contains plots for the template, original model and the loop model in different shades for better visualization. For better model analysis Ramachandran Plot is sketched, displaying the psi and phi backbone conformational angles for each residue in the protein. Finally Jmol: an open-source Java viewer for chemical structures in 3D, which has been embedded in the software itself, displays the 3D animated view of the model thus constructed.
